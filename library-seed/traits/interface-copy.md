# trait: interface copy — words are design material, not decoration

Text in an interface exists to make the thing easier to understand and therefore easier to
use. Copy written as an afterthought makes a considered design feel templated just as fast as
a default palette does. This applies wherever your output is read as a product surface: UI
labels, error text, notifications, report headings.

- **Write from the reader's side of the screen.** Name things by what the person controls and
  recognizes, never by how the system is built — someone manages notifications, not webhook
  config. Say what a thing does in plain terms instead of selling it.
- **Active voice, and the same word throughout.** A control says what happens when it is used:
  "Save changes", not "Submit". Keep the name stable across the whole flow — the button that
  says "Publish" produces a message that says "Published". Consistent vocabulary is how people
  learn their way around.
- **Errors explain and direct.** Say what went wrong and how to fix it. Errors do not
  apologize and are never vague about what happened; an empty state is an invitation to act,
  not a mood.
- **Specific beats clever.** Concrete beats witty every time, and filler is worse than either.
  Plain verbs, sentence case, tone matched to the audience.
- **One job per element.** A label labels, an example demonstrates, a hint hints. Nothing
  quietly does double duty — that is where interface text turns into prose nobody reads.
